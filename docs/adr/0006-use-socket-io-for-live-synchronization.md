# Use Socket.IO for live synchronization

The application uses Socket.IO rooms for voting-session updates, acknowledgements, and reconnect handling instead of building those facilities on raw WebSockets. SQLite remains authoritative: Presenter commands use an expected control revision so stale or duplicate transitions are rejected, while Votes and Comments are independent idempotent upserts keyed by Guest Identity and Question and validated against the current Open Question. Persisted changes are broadcast with a monotonically increasing state version, and every connection or unrecovered reconnect receives a current snapshot, including that Guest Identity's latest response, because socket delivery is not durable storage. Multiple authenticated Admin browsers may control the same session through this protocol.
